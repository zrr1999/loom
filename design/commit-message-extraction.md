# commit_message.py 是否拆分为独立仓库

## 结论

暂时不建议把 `src/loom/commit_message.py` 拆成新的仓库。

它现在更像是 `loom` 仓库里的一个配套集成点，而不是一个已经形成独立产品边界的通用组件。单独拆仓会增加发布、版本同步、文档维护和 hook 安装复杂度，但短期收益很有限。

## 当前耦合点

1. **入口直接绑定 `loom` 包名**
   - `prek.toml` 通过 `uv run python -m loom.commit_message` 调用它。
   - 一旦拆仓，当前 hook 入口、安装说明和本仓库开发环境都要一起调整。

2. **策略本身是仓库约定，不是通用标准库**
   - 允许的 type、emoji 前缀格式、以及 `fixup!` / `squash!` 例外都服务于当前仓库的提交规范。
   - README 和 `docs/reference/cli.md` 也把这套规则作为本项目工作流的一部分来描述。

3. **测试和发布链路依赖当前项目**
   - 测试位于 `tests/unit/test_commit_message.py`，跟随 `loom` 的 pytest/ruff/ty 配置一起运行。
   - 当前没有单独的版本边界、配置接口或独立分发需求。

## 拆仓成本

- 需要新建包名、发布流程和版本策略。
- 需要决定是做成通用 CLI、可导入库，还是 pre-commit/prek 专用 hook。
- 需要把当前 repo 的 hook 配置、README、参考文档和开发脚本一起迁移。
- 后续每次调整提交规范，都要跨仓库联动变更与升级版本。

## 可能收益

- 如果后续多个仓库都要复用同一套提交消息校验逻辑，拆分后可以集中维护。
- 如果计划把规则做成可配置产品（例如自定义 type 列表、模板、错误文案），独立仓库会更清晰。

## 建议路径

1. **现在先保留在本仓库内**，继续把它当作 `loom` 的仓库级工具。
2. **等出现明确复用信号再抽离**，例如：
   - 至少 2-3 个仓库要共享这套逻辑；
   - 需要脱离 `loom` 包名独立安装；
   - 需要对规则做可配置化而不是仓库内硬编码。
3. 如果未来要抽离，优先先做 **包内解耦**：
   - 把校验规则与 CLI 入口分层；
   - 把可配置项收敛成明确的数据结构；
   - 再决定是否升级成独立仓库。
