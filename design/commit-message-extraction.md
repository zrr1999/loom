# commit message 校验器是否拆分为独立仓库

## 结论

即使已经出现“其他仓库也想复用”这个信号，**现在依然不建议把 commit message 校验器立刻拆成独立仓库**，但也不应继续把它混在 `src/loom/` 主包里。

它现在更像是 `loom` 仓库里的一个配套集成点，而不是一个已经形成独立产品边界的通用组件。单独拆仓会增加发布、版本同步、文档维护和 hook 安装复杂度，但短期收益很有限。更合适的边界是：**继续保留在当前仓库的 repo-local tooling 下，例如 `tools/hooks/commit_message_validator.py`，并把“如何在别的仓库复用”写成明确规则，而不是提前拆仓。**

## 当前耦合点

1. **入口和安装仍然绑定当前仓库工作流**
   - `prek.toml` 通过 repo-local 命令触发这个校验器。
   - 说明它首先是当前仓库的开发工具，而不是独立分发的公共产品。

2. **策略本身是仓库约定，不是通用标准库**
   - 允许的 type、emoji 前缀格式、以及 `fixup!` / `squash!` 例外都服务于当前仓库的提交规范。
   - README 和 `docs/reference/cli.md` 也把这套规则作为本项目工作流的一部分来描述。

3. **测试和发布链路依赖当前项目**
   - 测试位于 `tests/unit/hooks/test_commit_message_validator.py`，跟随 `loom` 的 pytest/ruff/ty 配置一起运行。
   - 当前没有单独的版本边界、配置接口或独立分发需求。

4. **真正需要解耦的是包边界，而不是马上拆仓**
   - 之前的实现在 `src/loom/commit_message.py` 下，容易让人误以为这是 `loom` 运行时 API 的一部分。
   - 把它移动到 `tools/hooks/` 后，仍能复用同一仓库的测试/文档/Hook 配置，同时明确它只是仓库级辅助工具。

## 拆仓成本

- 需要新建包名、发布流程和版本策略。
- 需要决定是做成通用 CLI、可导入库，还是 pre-commit/prek 专用 hook。
- 需要把当前 repo 的 hook 配置、README、参考文档和开发脚本一起迁移。
- 后续每次调整提交规范，都要跨仓库联动变更与升级版本。

## 可能收益

- 如果后续多个仓库都要复用同一套提交消息校验逻辑，拆分后可以集中维护。
- 如果计划把规则做成可配置产品（例如自定义 type 列表、模板、错误文案），独立仓库会更清晰。

## 在“其他仓库也想复用”前提下的判断

- **“有人想复用”不等于“已经形成独立产品边界”**。当前校验器仍然绑定本仓库的提交约定、错误文案和 `prek.toml` 安装方式。
- 现在最现实的复用方式不是发布一个新仓库，而是把它当成一段**可复制、可局部定制、带测试样例的 repo-local hook**。
- 如果另一个仓库只是想采用同样的提交格式，当前建议是：
  1. 复制 `tools/hooks/commit_message_validator.py`；
  2. 一起复制 `tests/unit/hooks/test_commit_message_validator.py` 作为最小回归集；
  3. 在目标仓库自己的 `prek.toml` 或等价 hook 配置里接入；
  4. 再按该仓库自己的 type、scope 约定做局部调整。
- 这样做虽然有少量复制成本，但能保留各仓库的规范主权，也避免为了“也许以后会共享”而过早引入发布和版本治理负担。

## 建议路径

1. **现在先保留在本仓库内**，继续把它当作 `loom` 的仓库级工具，并维持当前 `tools/hooks/` + `tests/unit/hooks/` 的边界。
2. **把当前推荐复用路径文档化**：其他仓库如需采用同类规则，优先复制 validator、对应测试和 hook 配置，而不是立即依赖一个新仓库。
3. **等出现明确的“协同演进”信号再抽离**，例如：
   - 至少 2-3 个仓库需要共享同一份实现，并且更新时希望同步升级而不是各自修改；
   - 需要脱离当前 repo-local tooling 入口独立安装，例如要提供 `uvx`/pipx/独立 CLI；
   - 需要对规则做可配置化，而不是继续仓库内硬编码；
   - 需要独立版本节奏、兼容性承诺或跨仓库发布说明。
4. 如果未来要抽离，优先先做 **包内解耦**：
   - 把校验规则与 hook/CLI 入口分层；
   - 把可配置项收敛成明确的数据结构；
   - 再决定是否升级成独立仓库。
